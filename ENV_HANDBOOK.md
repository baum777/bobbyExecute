# BobbyExecution Environment Handbook

**Version:** Drafted from the provided `.env.example`  
**Audience:** Operators, developers, reviewers, and deployment owners  
**Primary goal:** Explain what each environment variable does, whether it is required, how sensitive it is, where to obtain any external key, and what operational behavior changes when it is set incorrectly.

---

## 1. Purpose of this handbook

This handbook is the operator-facing reference for BobbyExecution environment configuration. It is written to help you:

- understand what each variable controls,
- distinguish **hard secrets**, **provider API keys**, **runtime tuning knobs**, and **UI settings**,
- know which values are safe to leave empty,
- know which values must be generated locally,
- know which values must be obtained from an external provider dashboard,
- review staging and production readiness before deployment.

This document is intentionally conservative. When a setting can affect execution, safety, or operator control, the guidance assumes **fail-closed** behavior should be preferred.

## 3. Global safety rules

Before going section by section, keep these rules in mind:

### 3.1 Live execution should never be enabled accidentally

A safe baseline for local work is:

```env
LIVE_TRADING=false
DRY_RUN=true
TRADING_ENABLED=false
RPC_MODE=stub
```

### 3.2 Stub mode is for development and deterministic testing

If `RPC_MODE=stub`, you should treat the process as non-production and non-live.

### 3.3 Secrets must never be placed in browser-visible variables

Anything prefixed with `NEXT_PUBLIC_` is browser-visible and must never contain provider keys, tokens, or privileged control values.

### 3.4 Provider keys and local control secrets are different things

Do not confuse:
- a **third-party API key** such as `OPENAI_API_KEY`, and
- an **internal app secret** such as `CONTROL_TOKEN`.

Provider keys come from the provider. Internal secrets are generated by your team.

### 3.5 Empty is sometimes correct

Several integrations are designed to be optional. Leaving a value empty can be safer than inserting a dummy value that makes the system appear configured when it is not.

## 4. Mode and dependency summary

This is the fastest way to review whether a deployment is coherent.

| Goal | Minimum conditions |
|---|---|
| Safe local development | `LIVE_TRADING=false`, `DRY_RUN=true`, `TRADING_ENABLED=false`, `RPC_MODE=stub` |
| Paper-like real chain observation | `LIVE_TRADING=false`, `DRY_RUN=true`, `RPC_MODE=real`, valid `RPC_URL` |
| Live-test constrained trading | `LIVE_TRADING=true`, `DRY_RUN=false`, `TRADING_ENABLED=true`, `RPC_MODE=real`, wallet and control path configured, live-test limits set |
| Full live trading | `LIVE_TRADING=true`, `DRY_RUN=false`, `TRADING_ENABLED=true`, `RPC_MODE=real`, wallet and provider execution path configured |
| Operator-ready dashboard login | `DASHBOARD_SESSION_SECRET` and `DASHBOARD_OPERATOR_DIRECTORY_JSON` set on dashboard service |
| Operator-ready control mutations | matching `CONTROL_TOKEN` across control-facing participants |
| Restart orchestration | `CONTROL_RESTARTS_ENABLED=true` and valid `WORKER_DEPLOY_HOOK_URL` |
| External restart alerts | webhook URL configured for the selected alert destination |

## 5. External key acquisition guide

This section lists only the values that must be obtained from external platforms.

### 5.1 Solana RPC providers

#### `RPC_URL` via Helius
- Obtain from the Helius dashboard after creating an API key.
- Helius states that the same API key gives access to its RPC nodes and other Helius APIs.
- Typical format: `https://mainnet.helius-rpc.com/?api-key=YOUR_KEY`

#### `RPC_URL` via Alchemy
- Create an Alchemy app and use the generated Solana endpoint.
- Alchemy documents Solana endpoints in the `https://solana-mainnet.g.alchemy.com/v2/{apiKey}` format.

### 5.2 LLM providers

#### `XAI_API_KEY`
- Create in the xAI Console API Keys page.
- Export to `XAI_API_KEY` when using xAI-backed launch modes.

#### `OPENAI_API_KEY`
- Create on the OpenAI API platform at the project API Keys page.
- Prefer a dedicated project key with limited permissions.

### 5.3 Data and execution providers

#### `MORALIS_API_KEY`
- Create in the Moralis dashboard under Data APIs.
- Moralis documents API-key-based authentication for Data API usage.

#### `JUPITER_API_KEY`
- Create in the Jupiter Developer Portal / Developer Platform.
- Jupiter documents `x-api-key` header authentication for keyed endpoints.

### 5.4 Deployment platform

#### `RENDER_API_KEY`
- Create in Render Account Settings.
- Treat as highly privileged because it can provide broad platform access.

## 6. Local secrets you must generate yourself

These values are **not** obtained from a vendor and should be generated internally.

| Variable | How to generate | Sensitivity |
|---|---|---|
| `CONTROL_TOKEN` | Cryptographically random string | High |
| `OPERATOR_READ_TOKEN` | Cryptographically random string | High |
| `DASHBOARD_SESSION_SECRET` | Cryptographically random string | High |
| `NOTIFY_WEBHOOK_*_TOKEN` | Depends on your destination system | High |
| `CONTROL_RESTART_ALERT_WEBHOOK_TOKEN` | Depends on your destination system | High |

### Recommended generation rule
Use at least 32 random bytes rendered as hex or base64url for internal secrets unless the receiving service has stricter format expectations.

# 7. Section-by-section variable reference

## 7.1 Base environment

### `NODE_ENV`
- **Type:** enum string
- **Allowed values:** `development`, `test`, `production`
- **Required:** yes
- **Purpose:** Informs the application and dependency stack which runtime posture is active.
- **Recommended values:**
  - local dev: `development`
  - CI/tests: `test`
  - deployed services: `production`
- **Risk if wrong:** Logging, safety posture, framework defaults, and conditional behavior may not match the real runtime.

---

## 7.2 Execution mode – safety and trading control

### `LIVE_TRADING`
- **Type:** boolean
- **Required:** yes
- **Purpose:** Enables real execution mode.
- **Allowed values:** `true`, `false`
- **Recommended default:** `false`
- **Operational effect:** If `true`, the system should be treated as execution-capable and reviewed accordingly.
- **Risk if wrong:** May enable real trades unexpectedly.

### `DRY_RUN`
- **Type:** boolean
- **Required:** yes
- **Purpose:** Forces simulated behavior and prevents actual swaps.
- **Allowed values:** `true`, `false`
- **Recommended default:** `true`
- **Operational effect:** When `true`, it should override live intent and keep execution non-destructive.
- **Risk if wrong:** Setting `false` removes an important safety guard.

### `TRADING_ENABLED`
- **Type:** boolean
- **Required:** yes
- **Purpose:** Enables or disables the trading module at the feature level.
- **Allowed values:** `true`, `false`
- **Recommended default:** `false`
- **Operational effect:** Even if other live flags are set, disabling this should keep trading blocked.
- **Risk if wrong:** Feature exposure may not match the intended rollout stage.

### Review guidance
For most non-live reviews, keep the triple lock:

```env
LIVE_TRADING=false
DRY_RUN=true
TRADING_ENABLED=false
```

## 7.3 RPC and blockchain

### `RPC_MODE`
- **Type:** enum string
- **Allowed values:** `stub`, `real`
- **Required:** yes
- **Purpose:** Selects deterministic fake-chain behavior or real Solana RPC.
- **Recommended default:** `stub`
- **Production value:** `real`
- **Risk if wrong:** Live or staging behavior may silently use fake data or unintentionally hit production RPC.

### `RPC_URL`
- **Type:** URL string
- **Required:** mode-dependent
- **Required when:** `RPC_MODE=real`
- **Purpose:** Defines the actual Solana RPC endpoint.
- **Examples:**
  - public default: `https://api.mainnet-beta.solana.com`
  - Helius: `https://mainnet.helius-rpc.com/?api-key=YOUR_KEY`
  - Alchemy: `https://solana-mainnet.g.alchemy.com/v2/YOUR_KEY`
- **Risk if wrong:** Requests fail, rate limits appear unexpectedly, or traffic goes to an underpowered endpoint.

### Acquisition notes
- Use a managed provider for operational deployments.
- Keep provider-backed RPC URLs server-side only.

## 7.4 Stage 3 live-test mode

### `LIVE_TEST_MODE`
- **Type:** boolean
- **Required:** optional
- **Purpose:** Enables limited-capital live-test constraints.
- **Recommended default:** `false`
- **Operational effect:** Activates the special test-era budget controls listed below.

### `LIVE_TEST_MAX_CAPITAL_USD`
- **Type:** number
- **Required:** when live-test mode is used
- **Purpose:** Caps capital per trade during constrained live testing.
- **Example value:** `100`
- **Review guidance:** Keep low enough that one mistaken execution is operationally tolerable.

### `LIVE_TEST_MAX_TRADES_PER_DAY`
- **Type:** integer
- **Required:** when live-test mode is used
- **Purpose:** Limits trade count in the test window.
- **Example value:** `1`
- **Review guidance:** Start with one trade per day until observability proves stable.

### `LIVE_TEST_MAX_DAILY_LOSS_USD`
- **Type:** number
- **Required:** when live-test mode is used
- **Purpose:** Hardens maximum allowable daily loss.
- **Example value:** `50`
- **Review guidance:** Must be lower than the capital you are willing to burn during validation.

## 7.6 Wallet and core security

### `WALLET_ADDRESS`
- **Type:** base58 string
- **Required:** mode-dependent
- **Required when:** trading or live wallet-aware flows are active
- **Purpose:** Defines the trading wallet address.
- **Review guidance:** Verify the address against your wallet custody records before any live action.
- **Risk if wrong:** Funds, monitoring, or validation paths target the wrong wallet.

### `CONTROL_TOKEN`
- **Type:** secret string
- **Required:** service- and feature-dependent; effectively required for protected mutating control flows
- **Purpose:** Authenticates mutating control routes.
- **Generation:** generate locally; do not obtain from a vendor.
- **Important note:** If multiple services must mutually trust control assertions, they must use the same configured value where required.
- **Risk if wrong:** Mutations fail, or the wrong service can send privileged commands.

### `CONTROL_RESTART_ALERT_NOTIFICATION_COOLDOWN_MS`
- **Type:** integer milliseconds
- **Required:** optional
- **Purpose:** Rate-limits repeated restart alert notifications.
- **Example value:** `300000`
- **Operational effect:** Prevents alert spam during flapping conditions.

### `OPERATOR_READ_TOKEN`
- **Type:** secret string
- **Required:** feature-dependent
- **Purpose:** Authenticates protected operator read endpoints.
- **Generation:** generate locally.
- **Risk if wrong:** Read-only incident and runtime surfaces are inaccessible or insufficiently protected.

### `LOG_LEVEL`
- **Type:** enum string
- **Allowed values:** `debug`, `info`, `warn`, `error`
- **Required:** yes
- **Recommended default:** `info`
- **Operational effect:** Controls verbosity and noise level.
- **Review guidance:** Use `debug` briefly in staging, not as a long-term production default unless necessary.

## 7.7 Notification routing and restart alerts

This block defines how operational alerts are routed to external destinations.

### `NOTIFY_ROUTING_POLICY_MODE`
- **Type:** enum-like string
- **Required:** optional
- **Purpose:** Chooses between default routing logic and explicit destination selection behavior.
- **Typical value:** `default`
- **Risk if wrong:** Messages may go to the wrong destination set or not route as intended.

### Destination groups
Each destination is a full routing definition:
- Primary: production main escalation target
- Secondary: optional backup or separate escalation target
- Staging: optional non-production target

Each destination uses the same pattern.

#### `NOTIFY_WEBHOOK_<DEST>_URL`
- **Type:** URL string
- **Required:** only if that destination is enabled
- **Purpose:** External webhook endpoint.
- **Empty behavior:** destination is effectively unavailable.

#### `NOTIFY_WEBHOOK_<DEST>_TOKEN`
- **Type:** secret string
- **Required:** only if the destination expects authenticated requests
- **Purpose:** Credential sent to the webhook target.

#### `NOTIFY_WEBHOOK_<DEST>_HEADER`
- **Type:** string
- **Required:** optional
- **Purpose:** Header name used to deliver the token.
- **Typical value:** `authorization`

#### `NOTIFY_<DEST>_ENABLED`
- **Type:** boolean
- **Purpose:** Enables or disables that destination.

#### `NOTIFY_<DEST>_FORMAT`
- **Type:** string
- **Purpose:** Selects payload formatting mode.
- **Typical value:** `generic`

#### `NOTIFY_<DEST>_COOLDOWN_MS`
- **Type:** integer milliseconds
- **Purpose:** Rate-limits repeated notifications to that destination.

#### `NOTIFY_<DEST>_RECOVERY_ENABLED`
- **Type:** boolean
- **Purpose:** Allows sending recovery notifications when a prior failure state clears.

#### `NOTIFY_<DEST>_REPEATED_FAILURE_SUMMARY_ENABLED`
- **Type:** boolean
- **Purpose:** Controls whether repeated failures are summarized instead of always sent individually.

#### `NOTIFY_<DEST>_ALLOW_WARNING`
- **Type:** boolean
- **Purpose:** Controls whether warning-level events are eligible for delivery.

#### `NOTIFY_<DEST>_ENVIRONMENT_SCOPE`
- **Type:** string
- **Purpose:** Limits destination usage to a deployment environment such as `production` or `staging`.

#### `NOTIFY_<DEST>_PRIORITY`
- **Type:** integer
- **Purpose:** Orders destinations for routing precedence.

#### `NOTIFY_<DEST>_TAGS`
- **Type:** comma-separated string
- **Purpose:** Labels the destination for policy or diagnostic use.

### Legacy compatibility fallback

#### `CONTROL_RESTART_ALERT_WEBHOOK_URL`
- **Type:** URL string
- **Purpose:** Legacy primary fallback for restart alert delivery.
- **Use case:** Older deployments or compatibility retention.

#### `CONTROL_RESTART_ALERT_WEBHOOK_TOKEN`
- **Type:** secret string
- **Purpose:** Optional credential for the legacy alert webhook.

#### `CONTROL_RESTART_ALERT_WEBHOOK_HEADER`
- **Type:** string
- **Purpose:** Header name used for the legacy webhook token.
- **Typical value:** `authorization`

#### `CONTROL_RESTART_ALERT_WEBHOOK_TIMEOUT_MS`
- **Type:** integer milliseconds
- **Purpose:** Request timeout for legacy webhook dispatch.
- **Example value:** `5000`

#### `CONTROL_RESTART_ALERT_WEBHOOK_REQUIRED`
- **Type:** boolean-like policy flag
- **Purpose:** Intended to express requiredness at the configuration layer.
- **Review note:** Treat as a deployment policy indicator, not as proof of runtime enforcement, unless verified in code.

# 9. Common misconfiguration traps

## Trap 1: Live flags conflict with dry-run intent
If `LIVE_TRADING=true` but `DRY_RUN=true`, operators may assume live behavior while execution remains blocked. Keep the rollout state unambiguous.

## Trap 2: `RPC_MODE=real` with a weak or public shared RPC URL
The system may appear healthy under light load and then fail under real usage because the endpoint is rate-limited or unstable.

## Trap 3: Reusing one secret across environments
A staging leak becomes a production incident if the same control or session secret is reused.

## Trap 4: Enabling destination flags without a real webhook URL
Alert routing appears enabled on paper but silently has nowhere reliable to send events.

## Trap 5: Browser config confused with server config
Anything `NEXT_PUBLIC_*` is public. Never place provider keys or control tokens there.

## Trap 6: LLM model name drift
A model string that once worked may later be retired or renamed by the provider. Review model IDs during deployment updates.

## Trap 7: Dashboard auth half-configured
Setting only one of `DASHBOARD_SESSION_SECRET` or `DASHBOARD_OPERATOR_DIRECTORY_JSON` creates an unusable or misleading operator auth state.

## Trap 8: Legacy fallback variables treated as proof of runtime enforcement
A variable present in `.env.example` or deployment config is not proof that application code actively reads and enforces it.

# 10. Reviewer checklist

Use this checklist before promoting a configuration set.

## 10.1 Safety posture
- [ ] `LIVE_TRADING`, `DRY_RUN`, and `TRADING_ENABLED` form a coherent state
- [ ] `RPC_MODE` matches the intended deployment posture
- [ ] `MAX_SLIPPAGE_PERCENT` is explicitly reviewed
- [ ] `REVIEW_POLICY_MODE` is explicitly reviewed

## 10.2 Secrets and keys
- [ ] All internal secrets were generated locally
- [ ] No browser-visible variable contains a secret
- [ ] Staging and production do not reuse the same secrets
- [ ] Provider keys belong to the correct provider account/project

## 10.3 Operator control
- [ ] `CONTROL_TOKEN` is set where mutating control flows require it
- [ ] `OPERATOR_READ_TOKEN` is set if protected read routes are in use
- [ ] Dashboard auth is either fully configured or intentionally disabled

## 10.4 Notifications and restart handling
- [ ] Every enabled destination has a valid URL
- [ ] Cooldowns and priorities are intentional
- [ ] Staging and production routes are separated
- [ ] Legacy fallback variables are either intentionally used or clearly ignored

## 10.5 Deployment integration
- [ ] `RENDER_API_KEY` exists only if Render API automation is needed
- [ ] Restart orchestration settings match the real deployment pattern
- [ ] Journaling path is writable and retained
